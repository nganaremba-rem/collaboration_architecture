# 🪜 WARP セットアップ — ステップ・バイ・ステップ（クリック手順つき）

> **Cloudflare WARP** を **開発サーバー（GitLab ＋テストサイト）** 向けに設定する
> ための、初心者にもやさしい完全ガイドです。各クリックをそのまま追えるように
> 書いています。
>
> 🎯 **目的：** チームが **ログイン** で開発サーバーにアクセスできるようにする
> （IP アドレスではなく）。かつ **WARP から本番環境へ一切経路を作らない。**
>
> 触る場所は 3 つ：**AWS**、**開発サーバー（SSH 経由）**、**Cloudflare**。
> 上から順に進めてください。安全に関する注意は飛ばさないこと。

---

## ✅ はじめる前に

次を用意してください：

- [ ] **AWS コンソールのログイン**（開発サーバーのアドレス確認用）。
- [ ] Cloudflare アカウント作成用の **会社メール**（個人 Gmail は不可）。
- [ ] 開発サーバーへの **SSH アクセス**（ターミナルからログインできる状態）。
- [ ] 約 **30〜45 分**と、コーヒー ☕。

### ⚠️ 安全のための 2 つのルール（最初に一読）

1. **このガイドの間、AWS のポートは閉じないこと。** 現在のアクセスは予備として
   ずっと使えます。誰もロックアウトされません。
2. **AWS Session Manager を使える状態に保つこと** — 万一のときの緊急入口です。

> 🧭 **ダッシュボードの注意（最初に一読）：** Cloudflare は **Zero Trust** を
> **Cloudflare One** に改名し、設定の置き場所を大きく動かしました。左メニューは
> どちらの名前で表示されることもあります。大きな変更点：以前の
> **Settings → WARP Client** と **Settings → Authentication** のページは
> **なくなりました**。これらの設定は今 **Team & Resources → Devices** の下にあり、
> トンネルは **Networks → Connectors** の下にあります。本ガイドは **最新の** 経路を
> 使っています。メニュー名が少し違って見えても、**太字** の単語で照合してください
> — Cloudflare はラベルを頻繁に変えます。

## 🟦 パートA — 開発サーバーのプライベートアドレスを調べる（AWS）

開発サーバーの **プライベート IP**（AWS 内部のアドレス、例 `10.0.1.15`）が必要です。

1. **<https://console.aws.amazon.com>** にアクセスしてログイン。
2. 上部の検索バーに **EC2** と入力 → **EC2** をクリック。
3. 左メニュー → **インスタンス** をクリック。
4. 一覧から **開発サーバー（GitLab / テスト）** をクリック。
5. 下のパネルで **ネットワーキング** タブを開く。
6. **プライベート IPv4 アドレス** を確認。メモする。例 `10.0.1.15`。

> 📝 **ここに記入：** 開発プライベート IP = `________________`

✅ **表示されるはず：** `10.0.1.x` のような番号。これが WARP で開けるアドレスです。

---

## 🟦 パートB —（推奨）そのアドレスが変わらないようにする

AWS についての朗報：

- プライベート IP は、**再起動や停止／開始** では **変わりません**。
- **削除して作り直した** ときだけ変わります。
- （*パブリック* IP は停止／開始で変わりますが、WARP はそちらを使いません。）

つまり通常の再起動なら **すでに大丈夫** — プライベート IP は安定しています。

👉 開発サーバーを **頻繁に作り直す**（再起動ではなく）場合のみ、アドレスを完全に
固定するため次を実施：

1. EC2 → **インスタンス** → 開発サーバーをクリック。
2. **アクション → ネットワーキング → IP アドレスの管理**。
3. ネットワークインターフェイスで **プライベート IP を割り当て** を選び、
   **「自分で選択」** にチェックし、パートA でメモした正確なアドレスを入力。
4. **保存**。

✅ **表示されるはず：** 同じプライベート IP が割り当て済みと表示される。もうずれません。

## 🟦 パートC — Cloudflare アカウント ＋ Zero Trust チームを作成

1. **<https://dash.cloudflare.com/sign-up>** にアクセス。
2. **会社メール** でサインアップ → メールで確認。
3. ログイン後、左メニュー → **Zero Trust** をクリック。
4. **チーム名** を求められる → シンプルな名前、例 `yourcompany`。
   - これがログインページになる：`yourcompany.cloudflareaccess.com`。
   - ⚠️ これは **ログイン用ポータル名にすぎず**、実際のサイトのドメインでは
     **ありません**。DNS も変わりません。
5. **Free** プラン（最大 50 ユーザー、$0）を選択。確定。

> 💳 **注意：** このサインアップ時に、Cloudflare はプラン確認のため **支払い方法
> （クレジットカード、PayPal、または Google Pay）の登録を求めることがよくあります** —
> **無料（$0）プラン** でも同様です。登録しても **課金されません**。**Free** を
> 選んでいる限り、自分で有料に **アップグレードしない限り請求はありません**。
> カードは登録されるだけです。

✅ **表示されるはず：** Zero Trust ダッシュボードが開く。

## 🟦 パートD — ログイン方法を確認（ワンタイム PIN）

朗報：**ここでオンにする操作は不要です。** ワンタイム PIN は **既定** です —
ID プロバイダーを接続していない場合、各自はメールを入力してコードを受け取り
ログインします。この画面は **確認のために開くだけ** です。

1. Zero Trust → 左メニュー → **Team & Resources** → **Devices**。
2. ページ上部の **Management** タブをクリック（「Device profiles」の隣）。
3. **Device enrollment** セクションの **Device enrollment permissions** を見る。
   その場の説明に *"By default, users can log in with a one-time pin."*（既定で
   ワンタイム PIN でログイン可能）とあります — これが OTP 有効の証拠。続いて
   **Manage** をクリック。
4. **Login methods** タブを開く。**Authentication** パネルが表示されます（現在は
   **`New`** バッジと **Identity / MFA** タブ付き。Cloudflare が 2025 年中頃に
   この画面を刷新したため、以前の「one-time pin」の文言はここには表示されません）。

✅ **表示されるはず：** Authentication パネルで **"Accept all available identity
providers"** が **ON**。この ON 状態にワンタイム PIN が含まれるため OTP ログインは
有効です。**ここで変更は不要 — Save は押さないこと。**

> 💡 後で別のログイン方法（Google、Microsoft など）を足したい場合は
> **Integrations → Identity providers** で追加します — 本ガイドでは不要。
>
> 💡 この **Manage** 画面は開いたままに — **パートE** は隣のタブです。

## 🟦 パートE — 参加できる人を決める

無関係な人が端末を登録するのを防ぎます。

1. パートD と同じ場所：Zero Trust → **Team & Resources** → **Devices** →
   **Management** タブ → **Device enrollment permissions** → **Manage**。
2. **Policies** タブを開く。
3. **Create new policy（新しいポリシーを作成）** をクリック。
4. **ポリシー名：** `Company staff`。
5. **Action：** `Allow`。
6. **Include** で **Emails ending in（末尾が一致するメール）** → `@yourcompany.com`
   と入力。
   - （または **Emails** を選び、各メンバーのメールを貼り付け。）
7. **保存**。

✅ **表示されるはず：** 追加したポリシーが一覧に出る。指定メールのみ端末登録が可能に。
（ポリシーが 1 つも無いと誰も参加できません — 最低 1 つ必要です。）

## 🟦 パートF — 開発サーバーにトンネルを入れる

トンネルは小さなプログラム（`cloudflared`）で、サーバーから **外向き** に
Cloudflare へつなぎます。サーバー側に入口（受信ポート）は開きません。

1. Zero Trust → 左メニュー → **Networks** → **Connectors**。
2. **Add a tunnel（トンネルを追加）** をクリック。
3. **Cloudflared** を選択 → **Next**。
4. 名前を `dev-tunnel` にする → **Save tunnel**。
5. Cloudflare が **インストールコマンド** を表示。サーバーの OS（例 **Debian/Ubuntu
   64-bit**）を選び、表示されたコマンドを **コピー**。
6. ターミナルから **開発サーバーに SSH**：
   ```
   ssh your-user@<dev-server>
   ```
7. Cloudflare がくれたコマンド（`curl ...` または
   `cloudflared service install ...` で始まる）を **貼り付けて実行**。Enter。
8. Cloudflare の画面に戻り、トンネルが **Connected / 🟢 Healthy** になるまで待つ。

✅ **表示されるはず：** トンネルの横に 🟢 **Healthy**。（ならない場合は、正しい
サーバーでコマンド全文を貼ったか再確認。）

## 🟥 パートG — 開発サーバー「だけ」を経路にする（重要な分離ステップ）

ここが **本番を守る** ステップです。WARP に教えるのは **開発の 1 アドレスだけ** —
ネットワーク全体は絶対に教えません。

1. Zero Trust → **Networks** → **Routes**。
2. **CIDR** タブのまま → **Add a route** をクリック。
3. **CIDR** 欄に、開発 IP を **単一アドレス** として `/32` を付けて入力：
   ```
   10.0.1.15/32
   ```
   👉 パートA の **自分の** アドレス＋ **`/32`**。
4. （任意）**Description：** `dev server`。
5. **Tunnel** のドロップダウンで `dev-tunnel` を選ぶ。
6. **保存**。

> 🛑 **`10.0.1.0/24` のような全範囲は入れないこと。** それは近所一帯を開けてしまい
> ます。トンネルは **与えられた範囲内のどのホストにも到達できる** ため、同じ
> ネットワークにある **本番サーバー** にも WARP ユーザーが届いてしまいます。
> `/32` は「この 1 台だけ」という意味です。

✅ **表示されるはず：** `/32` で終わるエントリが **1 件だけ**。それより広いものは無し。

---

## 🟦 パートH — その address を WARP に実際に運ばせる

WARP は既定で `10.x` のようなプライベートアドレスを **無視** します。開発用の 1
アドレスだけ通します。

1. Zero Trust → **Team & Resources** → **Devices** → **Device profiles** タブ。
2. **General profiles** の下で、プロファイル名 **Default** をクリック → スライド
   表示されるパネルで **Edit** をクリック。
3. **Split Tunnels** セクションまでスクロール。ラジオボタンが 4 つあります：
   - **「Exclude IPs and domains」が選択（既定）の場合：**
     **Manage** をクリックし、自分のアドレスを含むエントリ（例 `10.0.0.0/8`）を
     見つけて **それだけ削除** — または **Include** モードに切り替える（次の選択肢）。
   - **「Include IPs and domains」を選ぶ場合：**
     **Manage** をクリックし、広い既定値を削除して、開発アドレス
     `10.0.1.15/32` **だけ** を **Add**。
4. **保存**（プロファイルページの上部または下部）。

> 💡 一番安全で簡単なのは **Include** モードに切り替え、`/32` の開発アドレス
> **だけ** を追加すること。WARP はそれ *だけ* を運び、他のプライベートは運びません。

✅ **表示されるはず：** Include に開発 `/32` アドレスがある、または Exclude から広い
プライベート範囲が消えている。

---

## 🟥 パートI —（追加の安全策）本番を明示的にブロック

念のための二重対策：本番への到達を **拒否** するルールを足し、将来のミスでも
本番に届かないようにします。

1. Zero Trust → **Traffic policies** → **Firewall policies**。
2. **Network** タブを開く（DNS / HTTP の隣）→ **Add a policy** をクリック。
3. **名前：** `Block production`。
4. **Action：** `Block`。
5. ルール：**Destination IP** → **in** → **本番** のアドレスまたは範囲
   （例 `10.0.2.0/24` ＝本番サブネット）を入力。
6. **保存** 後、ポリシーの **Status** トグルが **On** であることを確認（許可ルール
   より **上** に置く — ルールは上から順に評価されます）。

✅ **表示されるはず：** `Block production` ポリシーが一覧にあり、有効。

> 🔎 本番のアドレスが不明なら、パートA と同じ手順で調べる（EC2 → 本番インスタンス
> → ネットワーキング → プライベート IPv4）。そのサブネットを使う。

## 🟦 パートJ — ラップトップに WARP を入れてログイン

各メンバーの PC で実施。

1. **<https://1.1.1.1/>** にアクセスし **Cloudflare WARP**（Windows / macOS）を
   ダウンロード。
2. インストールしてアプリを **開く**。
3. **歯車 ⚙️（設定）** → **Account** → **Login with Cloudflare Zero Trust** をクリック。
4. パートC の **チーム名**（例 `yourcompany`）を入力。
5. ブラウザが開く → **会社メール** を入力 → メールで **コード** が届く → 入力。
6. アプリに戻り WARP を **On** にする。

✅ **表示されるはず：** WARP が **Connected** と表示。

## 🟩 パートK — テスト（動作確認 ＋ 本番が安全なことの確認）

**登録済みのラップトップ**（WARP Connected）から：

1. ブラウザで `https://10.0.1.15`（自分の開発 IP）を開く → **GitLab / テストが開く** ✅
2. 開発サーバーへ `git clone` / `git push` → **動く** ✅
3. **本番の予約システム** を通常のリンクで開く → **これまで通り開く** ✅
   （WARP はブロックしません）

次に安全の境界を確認：

4. **本番** のプライベートアドレス（例 `https://10.0.2.20`）に到達しようとする →
   **つながらないはず** ✅（パートG の `/32` ＋ パートI の拒否ルールで遮断）
5. **WARP なし／未ログイン** の PC で開発 IP を試す → **到達できない** ✅
6. 登録済み PC でスマホのテザリング（別ネットワーク）にする → 開発は依然動く ✅
   （IP に依存しなくなった証拠）

✅ 6 つすべて一致すれば完了です。

## 🟦 パートL — 後からの強化（任意・準備ができたら）

全員が WARP に十分慣れてから：

- 🔒 **古い入口を閉じる：** AWS のセキュリティグループで開発サーバーの受信
  `80/443/22` を閉じ、古い IP 許可ルールを削除。
  先に **AWS Session Manager** を緊急入口として確保しておくこと。
- 🛡️ **MFA を追加：** パスキー / Touch ID / Windows Hello（無料）を必須にする、
  または Google ログインを追加して一段強くする。

## 🆘 うまくいかないとき

- **WARP 接続後に開発が開かない：** たいていパートH が原因 — アドレスがまだ除外
  されています。Split Tunnel を **Include** にして `/32` を追加。
- **WARP にログインできない：** パートE に自分のメールが含まれているか、パートJ の
  チーム名が正確かを確認。
- **本番に到達できてしまう（本来ダメ）：** パートG が `/32`（`/24` でない）か、
  パートI のブロックポリシーが有効で許可ルールより上にあるか再確認。
- **サーバーからロックアウトされた：** **AWS Session Manager** を使う（残しておいた
  理由がこれ）。
