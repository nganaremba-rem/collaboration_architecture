# 🔌 Cloudflare WARP の仕組み — やさしい言葉で

> WARP 設定の後、みんなが GitLab とテストサイトにどうやって入るのかを、
> 専門用語なしで説明します。上から順に読むと分かりやすいです。

---

## 🎯 解決したい問題

- サーバーは Amazon（AWS）にあります。今はみんな **IPアドレス**（＝ネット上の住所）で
  サーバーにアクセスしています。
- でもその住所は**しょっちゅう変わります**。チームの住所は移動するたびに変わり（自宅・
  会社・カフェ・スマホ）、サーバーの住所も再起動で変わることがあります。「この住所だけ
  許可」では、すぐ使えなくなります。
- かといって、誰の住所でも入れるよう開けっぱなしにするのは危険です。

**考え方:** 「**住所**」を信じるのをやめ、「**人**」を信じる。会社のメールで一度ログイン
すれば、どこからでもサーバーに入れる。

---

## 🏠 たとえ話（これを覚えれば後は簡単）

サーバーを、**公開ドアのない、鍵のかかった倉庫の中のお店**だと想像してください。

- 🚇 **トンネル（`cloudflared`）** — サーバーに入れる小さなプログラム。**内側から外へ向けて
  専用の通路を掘り**、見張り付きの検問所までつなぎます。内側から掘るので、倉庫に公開ドアは
  要りません。
- 🛂 **Cloudflare** — **見張り付きの検問所**。世界中に支所があるので、必ず近くに1つあります。
  ID を見せないと誰も通れません。
- 📞 **WARP アプリ**（各自のPCに入れる）— あなたのPCを同じ検問所につなぐ**専用電話線**。
- 🪪 **ログイン（メール＋PIN）** — 検問所で**身分証を見せる**こと。

ID が確認できると、検問所が**あなたのPCの線**と**お店の通路**をつなぎます。すると、実際に
どこにいても、まるで倉庫の中に立っているかのようにお店とやり取りできます。

---

## 🧩 4つの部品

| 部品 | 何か | どこにある |
|---|---|---|
| **`cloudflared`（トンネル）** | サーバーを外の Cloudflare へつなぐ小さなプログラム | 各サーバー |
| **Cloudflare** | 真ん中にある世界規模の検問所／交換台 | インターネット上（Cloudflare の網） |
| **WARP アプリ** | 人のPCを Cloudflare につなぐアプリ | 各利用者のPC |
| **ログイン（ワンタイムPIN）** | メール＋コードで本人確認 | 1セッションに1回 |

---

## 🚶 実際に起きること（順番）

仲間が GitLab を開くとき:

1. **サーバーは既に通路を掘ってある。** `cloudflared` が Cloudflare へ外向きの接続を
   保ち続けています（一度設定すれば、自動で再接続）。
2. **利用者が WARP をオンにしてログイン。** 会社メールを入力 → コードがメールで届く →
   入力。✅ 本人確認 OK。
3. **PC が検問所につながる。** その瞬間、PC は会社の私的ネットワークの中にいるかのように
   振る舞います。
4. **私的アドレス（または内部の名前）で GitLab を開く。** 通信はこう流れます:
   PC → WARP → 一番近い Cloudflare 検問所 → サーバーの通路 → GitLab。
5. **GitLab が同じ道で返事する。** 利用者にはただの普通のサイトに見えます。いつも通り
   閲覧・`git push`・`git pull` ができます。

---

## 💡 なぜ「住所が変わる問題」が解決するのか

- **住所ではなく、ログインで本人確認するから。** IP が毎日変わっても、検問所は「あなた」を
  確認するので、ちゃんと入れます。
- **サーバーの住所も変わってよい。** サーバーは**自分の外向き通路**で検問所につながるので、
  公開IPが何であっても関係ありません。さらに WARP はサーバーの**私的アドレスの「町内全体
  （サブネット）」**を指すので、サーバーの私的住所が多少ずれても見つけられます。
- **再起動しても通路は掘り直される。** `cloudflared` は常駐サービスで、自動的に再接続します。

---

## 🔒 なぜ安全なのか

- サーバーは**公開ドアを一切開けません。** 入口は検問所だけで、検問所は ID を要求します。
  ネットを当てもなく探す人には、ノックするドアが見つかりません。
- 後で**古いドアを完全に施錠**（AWS のポートを閉じる）すれば、入口は検問所**だけ**になります。
- （今は切替で何も壊さないよう、古いドアは開けたまま。両方の道が並行して使えます。）

---

## ⚖️ 良い点・注意点（正直に）

- 👍 50人まで無料。ドメインや DNS の変更なし。100MB のアップロード上限もなし（大きな
  ファイルや Docker イメージも問題なし）。
- 👎 各自が **WARP アプリを一度入れて**ログインする必要あり（ブラウザだけでは不可）。
  その後はほぼ意識せず使えます。
- 👎 アクセスが Cloudflare の稼働に依存します（どんなサービスも1つに頼るのと同じ）。

---

## 🧠 ひとことまとめ

> サーバーは静かに**外へ**世界規模の検問所につながり、各自はPCを同じ検問所につないで
> ID を見せる。検問所が両者をつなぐ —— だから、**住所が変わっても、どこからでも、
> 正しい人だけがサーバーに入れる。しかもサーバーは外に晒されない。**

# ⚙️ 設定手順（やり方）

> 前提:Cloudflare 用の会社メール、両サーバーへの SSH + sudo、サーバーの私的IP/サブネット、
> 許可するメール一覧が既に手元にあること。

> ⚠️ **2つの安全ルール:**(1) 設定中は AWS のポートを閉じない（旧アクセスを保険に残す）。
> (2) ポートが開いていれば従来どおり SSH で入れるので、締め出される心配なし。

## ステップ1 — Cloudflare アカウントと Zero Trust チーム作成
1. <https://dash.cloudflare.com/sign-up> で **会社共有メール**（個人 Gmail 不可）で登録。
2. 左メニューの **Zero Trust** を開く。
3. **チーム名**を決める → `your-team.cloudflareaccess.com` になる（ログイン用の入口で、
   **本物のドメインではない**）。
4. **Free** プランを選ぶ（50人まで、$0）。

> 💡 この手順ではドメイン追加もネームサーバー変更も一切しません。

## ステップ2 — ログイン方法の設定（ワンタイムPIN）
1. Zero Trust → **Settings → Authentication**。
2. **One-time PIN** がオンか確認（初期状態でオン）。利用者は会社メールを入れてコードを受け取る。

## ステップ3 — WARP を入れてよい人を決める
1. Zero Trust → **Settings → WARP Client → Device enrollment permissions →
   Manage / Add a rule**。
2. **Action: Allow**、**Include → Emails ending in** `@company.com`（または一覧を貼付）。

## ステップ4 — 各サーバーでトンネルを動かす（私的ネットワークとして）
1. Zero Trust → **Networks → Tunnels → Create a tunnel → Cloudflared**。
2. 名前を付ける（例:`aws-private`）→ **Save**。
3. **インストールコマンド**をコピー → **サーバーに SSH** して実行。**Connected / Healthy**
   になるまで待つ（**外向き接続だけ** — 受信ポート不要）。
4. トンネルを開く → **Private Network** タブ → **Add a private network** → サーバーの
   **私的CIDR**（例:`10.0.1.0/24`）を入力。IP が変わっても効くよう、単一IPでなく
   **サブネット**を指定する。
5. `cloudflared` が**起動時に自動で立ち上がる**こと（サービスとして入る）を確認。毎日の
   再起動後に自動で再接続する。

## ステップ5 — WARP に私的IPを通させる（重要な落とし穴）
WARP は初期状態で私的アドレス帯を**除外**するので、サーバー帯を通す:
1. Zero Trust → **Settings → WARP Client → Split Tunnels**。
2. **Exclude（除外）**モードなら、サーバー帯を含む項目（例 `10.0.0.0/8`）を**削除**。
   または **Include** モードに切替え、サーバーの CIDR だけ追加。

## ステップ6 — 利用者の端末に WARP を入れて登録
1. **Cloudflare WARP**（1.1.1.1 アプリ）を入れる:<https://1.1.1.1/>（Windows/macOS）。
2. 開く → **Settings → Account → Login with Cloudflare Zero Trust**。
3. **チーム名**を入力 → ブラウザが開く → **会社メール** → メールの **PIN** を入力。
   WARP が **Connected** になる。

## ステップ7 — テスト
1. 登録済みPCで `https://<GitLabの私的IP>` を開く → GitLab が表示。
2. HTTPS か SSH で `git clone` / `push` → 動く（100MB上限なし）。
3. WARP を入れていないPC → 到達できない。
4. **別のネットワーク**（スマホのテザリング）からログイン → それでも入れる
   （住所に依存しない証明）。

## ステップ8 —（任意）生IPの代わりに名前を使う
`10.0.1.15` でなく `gitlab` と打てるように:
- **内部DNSレコード**（Route 53 プライベートホストゾーン）を追加、または
- Zero Trust **Gateway → Resolver / local domains** を使う。
サーバーの私的IPが変わっても利用者を守れる。

## ステップ9 —（任意）誰がどのサーバーに入れるか制限
- Zero Trust → **Gateway → Firewall policies → Network** → ユーザーグループ + 宛先
  IP/ポートで許可・拒否。

## ステップ10 — 後の強化（準備ができたら）
- 🔒 全員が WARP を使い始めたら、受信ポート（`80/443/22`）を**閉じ**、古いIP許可ルールを
  削除。閉じる**前に**緊急経路（AWS Session Manager）を用意する — 必要になるのはポートを
  閉じた後だけ。
- 🛡️ **MFA 追加:** passkey / Touch ID / Windows Hello を必須に（無料）、または Google
  ログインで SSO。

## ✅ 最終チェックリスト
- [ ] 公開サイト・メールに影響なし（DNS を触っていない）。
- [ ] トンネルが **Healthy**、私的サブネット追加済み、`cloudflared` 自動起動。
- [ ] 登録＆ログイン済みPCで GitLab・テストサイトに入れる。
- [ ] 未登録PCでは入れない。
- [ ] 別ネットワーク / 変わった IP でも動く。
- [ ] `git push` / `pull` が動く（大きいファイルもOK）。
